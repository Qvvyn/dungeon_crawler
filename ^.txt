 "Most resource-intensive combat visualization?"

  In this stack (Label3D billboards in a SubViewport + ASCII post-shader), the per-frame cost lives in the entity sync
  loop (FirstPersonRig.gd:810-1150), not the shader. Ranked worst→meh for live combat:

  1. Multi-line status-stacked enemies — a wizard/boss (5-row art) on fire + shocked + poisoned drives live_text to 8
  rows. Each row is a separate Label3D child whose .text / .modulate / .position write every frame, and the child count
  grows dynamically as statuses pile on. A roomful of these is the worst case the loop sees.
  2. Magma Boss arena with FirePatch trails — each FirePatch is a multi-line animated entity. Many patches = many
  simultaneous per-row Label3D writes. (FirePatch.gd was just heavily updated in v0.4.0.)
  3. Concurrent beam-sweep emitters — each enemy beam pools Label3D glyphs along its path. 3-5 sweepers firing at once =
   50-100 pooled Label3Ds positioned + oriented every frame.
  4. Floating damage text storm — nova / shotgun / pierce land multiple hits per frame; each spawns a Label that's
  re-projected from world to screen every other frame. Linear with hit count.
  5. Nova detonation — momentary spike: ~16 nova_shard projectiles + the animated +↔x core all register Label3Ds at
  once.

  Cheap by comparison: walls (one draw call per 8×8 chunk), single OmniLight3D (one), HP bars (text cached, modulate
  cached now), camera shake (one transform mul), occlusion (DDA staggered every 3 frames per entity).

  If you wanted to profile the worst case: start a Magma boss arena (biome 3), let it spawn enemies, freeze a few with
  the freeze gun, then enflame them. That hits multi-line + status stacking + persistent ice cubes + the boss's eruption
   FirePatches simultaneously.
